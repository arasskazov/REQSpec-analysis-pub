### ЭТАП 2: Содержательная проверка

#### Блок 2.1: Логическая цепочка целей

**Промпт**:
```text
БЛОК 2.1 — ПРОВЕРКА ЛОГИКИ "ЦЕЛИ → ЗАДАЧИ → РЕЗУЛЬТАТЫ → КРИТЕРИИ"

Проверь наличие и связность логической цепи в ТЗ.

ШАГ 1: Построй таблицу связей:

| Цель из ТЗ (цитата) | Раздел | Задачи, вытекающие из цели | Измеримые результаты | Критерии приёмки | Оценка связности (0-5) |
|--------------------|--------|---------------------------|---------------------|-----------------|---------------------|
| | | | | | |

ШАГ 2: Для каждой цели проверь:
- Есть ли конкретные задачи?
- Сформулированы ли измеримые результаты?
- Определены ли критерии достижения?

ШАГ 3: Выяви "цели-сироты" (без задач) и "результаты-сироты" (без критериев).

ШАГ 4: Проверь соответствие целей ТЗ целям из паспорта проекта (если загружен).

ФОРМАТ ОТВЕТА: Таблица + список разрывов логической цепи с указанием разделов ТЗ.
```

#### Блок 2.2: Измеримость требований

**Промпт**:
```text
БЛОК 2.2 — ПРОВЕРКА ИЗМЕРИМОСТИ ТРЕБОВАНИЙ

Найди субъективные и неизмеримые формулировки в требованиях.

ШАГ 1: Найди ВСЕ требования с маркерами субъективности:

Маркеры субъективности:
- "удобный", "простой", "интуитивный"
- "достаточный", "приемлемый", "оптимальный"
- "быстро", "эффективно", "качественно"
- "при необходимости", "по возможности"

Результат в таблице:

| ID требования | Раздел ТЗ | Субъективная формулировка (цитата) | Тип проблемы | Предлагаемый измеримый критерий |
|--------------|-----------|-----------------------------------|-------------|-------------------------------|
| | | "удобный интерфейс" | Субъективность | "Время выполнения типовой операции X ≤ 30 сек; не более 3 кликов для операции Y" |

ШАГ 2: Проверь требования на соответствие шаблону SMART:
- Specific (конкретное)
- Measurable (измеримое)
- Achievable (достижимое)
- Relevant (релевантное)
- Time-bound (ограниченное по времени)

ШАГ 3: Для нефункциональных требований проверь наличие числовых показателей:
- Производительность (операций/сек, время отклика)
- Надежность (доступность %, MTBF)
- Масштабируемость (количество пользователей, объём данных)

ФОРМАТ ОТВЕТА: Таблица субъективных формулировок + список требований без числовых критериев.
```

#### Блок 2.3: Полнота функциональных требований

**Промпт**:
```text
БЛОК 2.3 — ПРОВЕРКА ПОЛНОТЫ ФУНКЦИОНАЛЬНЫХ ТРЕБОВАНИЙ

Оцени покрытие бизнес-процессов функциональными требованиями.

ШАГ 1: Составь список критических процессов для данного типа системы (используй загруженный контекст отрасли).

Для PLM автопрома это:
- Управление составом изделия (BOM) с вариативностью
- Управление инженерными изменениями (ECR/ECO/ECN)
- Интеграция с CAD-системами
- Управление конфигурациями
- Миграция данных из legacy-систем
- Управление жизненным циклом изделия
- Управление документацией (ЕСКД/ЕСПД)

Результат в таблице:

| Критический процесс | Наличие в ТЗ (Да/Нет/Частично) | ID требований | Оценка детализации (0-5) | Пробелы |
|--------------------|-------------------------------|---------------|------------------------|---------|
| | | | | |

ШАГ 2: Для каждого процесса с оценкой < 3 укажи конкретные недостающие функции.

ШАГ 3: Проверь наличие требований к граничным/исключительным ситуациям:
- Обработка ошибок
- Откат операций
- Работа при частичной недоступности компонентов

ФОРМАТ ОТВЕТА: Таблица покрытия процессов + список критических пробелов.
```

#### Блок 2.4: Дублирование и противоречия

**Промпт**:
```text
БЛОК 2.4 — ПОИСК ДУБЛИРОВАНИЯ И ПРОТИВОРЕЧИЙ

Выяви дублирующиеся и противоречивые требования.

ШАГ 1: Поиск прямых дубликатов:
- Найди требования с идентичным или очень похожим текстом
- Проверь, не повторяются ли требования в разных разделах

Результат:

| ID требования 1 | ID требования 2 | Раздел 1 | Раздел 2 | Тип дубликата (прямой/смысловой) | Формулировка |
|----------------|----------------|----------|----------|--------------------------------|--------------|
| | | | | | |

ШАГ 2: Поиск смысловых дубликатов:
- Требования, описывающие одну функцию разными словами
- Требования, пересекающиеся по смыслу на 70%+

ШАГ 3: Поиск противоречий:
- Взаимоисключающие требования
- Требования, конфликтующие по ресурсам/производительности
- Несовместимые технологические решения

Пример: "высокая производительность обработки 3D" vs "использование interpreted languages для обработки геометрии"

ФОРМАТ ОТВЕТА: Две таблицы (дубликаты + противоречия) с конкретными ID требований.
```

#### Блок 2.5: Нефункциональные требования

**Промпт**:
```text
БЛОК 2.5 — ПРОВЕРКА НЕФУНКЦИОНАЛЬНЫХ ТРЕБОВАНИЙ

**КОНТЕКСТ**: Нефункциональные требования должны соответствовать 
ISO/IEC 25010. Этот стандарт определяет 8 характеристик качества ПО.

**⚠️ ВАЖНО**: Используй ТОЛЬКО категории из ISO/IEC 25010. 
Не добавляй вымышленные категории.

---

### ШАГ 1: Проверь требования по ISO/IEC 25010

Заполни таблицу для каждой из 8 характеристик ISO/IEC 25010:

| Характеристика ISO 25010 | Наличие в ТЗ (Да/Частично/Нет) | Раздел ТЗ | Измеримые метрики | Оценка (0-5) |
|--------------------------|--------------------------------|-----------|-------------------|--------------|
| Performance efficiency   |                                |           |                   |              |
| Reliability              |                                |           |                   |              |
| Security                 |                                |           |                   |              |
| Maintainability          |                                |           |                   |              |
| Compatibility            |                                |           |                   |              |
| Usability                |                                |           |                   |              |
| Portability              |                                |           |                   |              |
| Scalability              |                                |           |                   |              |

---

### ШАГ 2: Топ-5 критических недостающих требований

**ОГРАНИЧЕНИЯ**:
- Включи ТОЛЬКО требования из ISO/IEC 25010
- Максимум 5 критически важных недостающих требований
- Для каждого требования укажи пункт ISO/IEC 25010

**Формат таблицы**:

| № | Характеристика ISO 25010 | Недостающее требование | Предлагаемая метрика | Ссылка на ISO 25010 |

| 1 | Performance efficiency  | Время отклика UI       | ≤ 2 сек на операцию | ISO 25010, п. 8.1.2 |

| ... | ... | ... | ... | ... |

**Если в ТЗ уже есть все критичные требования** — укажи: 
"Критически важные НФТ присутствуют, дополнений не требуется."

---

### ШАГ 3: Конфликты между НФТ

Найди противоречия между требованиями (если есть):

| Требование 1 | Требование 2 | Конфликт | Рекомендация |
|-------------|-------------|----------|--------------|
| ... | ... | ... | ... |

**Если конфликтов нет** — укажи: "Конфликты НФТ не обнаружены."

---

### ФОРМАТ ОТВЕТА:
- Таблица из Шага 1 (полная)
- Таблица из Шага 2 (максимум 5 пунктов)
- Таблица из Шага 3 (если есть конфликты)

**НЕ ДОБАВЛЯЙ**:
- Требования вне ISO/IEC 25010
- Вымышленные стандарты
- Более 10 рекомендаций
```
